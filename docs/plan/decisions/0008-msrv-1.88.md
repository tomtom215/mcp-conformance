# ADR 0008: MSRV 1.88 — rmcp's Real Compilation Floor

**Date:** 2026-06-10
**Status:** Accepted (supersedes the MSRV clause of [ADR-0004](0004-toolchain-and-msrv.md))
**Author:** Tom F.

---

## Context

ADR-0004 set MSRV 1.85 (the Edition 2024 floor) and pre-committed to this moment:
"When rmcp lands (M2), its real compilation floor is validated against 1.85 and this
ADR is superseded if a bump is forced."

The validation happened on 2026-06-10, before any M2 server code. rmcp declares no
`rust-version` at any level — workspace manifest, crate manifest, README, all checked
at tag `rmcp-v1.7.0` ([register 3.5](../01-ecosystem-context.md)) — so the floor had
to be measured, not read:

- `cargo +1.85 check` against pinned rmcp `=1.7.0`, `=1.6.0`, `=1.5.0`, `=1.4.0`,
  `=1.2.0`, `=1.0.0`, and `=0.17.0` **all fail** with E0658: rmcp's own library
  source uses let-chains, stabilized in **Rust 1.88.0** (2026-06 verification against
  the [1.88.0 announcement](https://blog.rust-lang.org/2025/06/26/Rust-1.88.0/);
  Edition-2024-only feature).
- Independently, `rmcp-macros 1.7.0` requires `darling = "0.23"`, whose only release
  0.23.0 declares `rust-version: 1.88.0` — the proc-macro path floors at 1.88 even
  if the library compiled.
- `cargo +1.88 check` with the feature set M2 needs (`server`, `macros`,
  `transport-io`, `transport-streamable-http-server`) succeeds.

So every rmcp the M2 everything-server could be built on requires rustc ≥ 1.88, and
the choice ADR-0004 deferred is now forced: bump, split, or drop rmcp.

## Decision

1. **MSRV 1.88 workspace-wide**, in `[workspace.package] rust-version`, effective for
   every published crate. Uniform rather than split (libraries at 1.85, server at
   1.88) because the installed base is zero — all four crates first published
   2026-06-10 with no downloads — so a uniform bump costs nothing today, while
   split-MSRV costs CI-matrix and documentation complexity forever.
2. **Per ADR-0004's own policy, the bump is a minor release**: the next release is
   **0.2.0** with a changelog entry saying so explicitly.
3. Everything that encodes the MSRV moves in the same commit: CI matrices
   (`ci.yml`), release gates (`release.yml`), `clippy.toml` `msrv` hint,
   CONTRIBUTING. The dual-toolchain gate policy itself is unchanged — clippy and
   tests run on {stable, 1.88} exactly as they ran on {stable, 1.85}.
4. **Upstream follow-through** (engagement backlog item 3): propose to
   modelcontextprotocol/rust-sdk that rmcp declare `rust-version = "1.88"`. The
   measurement above is the evidence: undeclared-but-real MSRVs defeat cargo's
   MSRV-aware resolver for every downstream 1.85–1.87 user (resolution succeeds,
   compilation fails with E0658 — exactly what we hit). Nothing is posted without
   owner sign-off.

## Consequences

### Positive

- rmcp 1.7.0 (current) is usable with its macro support; M2 proceeds on the
  current SDK line, not a pinned museum piece.
- The MSRV claim stays tested-not-aspirational: CI gates every PR on 1.88 across
  three platforms, same shape as before.
- 1.88 (released 2025-06-26) is ~12 months old at adoption — still well inside
  ADR-0004's "toolchains lag in other projects' CI" tolerance, and five minors
  below the a2a-rust baseline (1.93).

### Negative

- Adopters pinned to 1.85–1.87 toolchains lose access to releases after 0.1.0.
  Mitigation: 0.1.0 remains published and functional; no known adopter exists yet
  (zero downloads at decision time).
- let-chains and other 1.86–1.88 std/syntax APIs become available to our own code;
  consistency with the "work around or wait" stance in ADR-0004's negatives now
  reads as "may use freely" — reviewers should not demand 1.85 workarounds anymore.

## Alternatives Considered

### Split MSRV: libraries 1.85, server crates 1.88

Workable today (`rust-version` is per-crate overridable), rejected: every
workspace-wide CI invocation (`cargo +MSRV clippy/test --workspace`) would need
exclude lists, the README claim becomes a footnote table, and the benefit — a 1.85
floor for `mcp-conformance-core` consumers — protects an installed base that does
not exist.

### Pin a pre-1.0 rmcp that compiles on 1.85

Empirically dead: 0.17.0 (2026-02-27, the last pre-1.0) already fails with the same
E0658. Older versions predate the streamable-HTTP server API maturity M2 needs, and
building the parity artifact against an SDK line upstream no longer ships would
invert the milestone's purpose.

### Drop rmcp; hand-roll the server on tokio/axum

Maximal control, no MSRV pressure — and a milestone re-plan: the roadmap's M2 DoD
says "on rmcp" because the parity story (an everything-server the rust-sdk ecosystem
can recognize as idiomatic) is the point. Rejected as out of proportion to a
12-month-old toolchain requirement.
